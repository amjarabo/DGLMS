⚡ How to use this playbook

Each major DGLMS workflow below has three path types: Happy (ideal flow), Sad (valid user, expected friction), and Negative (invalid/unauthorized intent).

Path content is environment-agnostic — the same scenarios run across all six environments per the testing protocol: DOT Dev → DOT QA → DOT UAT (DOT acceptance chain) and DTC QA → DTC UAT → Production (system integrator chain). Defects, sign-off, and results are tracked separately per environment in the E2E Dashboard.

🟢 Happy Path Valid user, valid data, expected system behavior. Demonstrates the feature works as designed end-to-end.
🟡 Sad Path Valid user encounters expected friction: missing field, validation rule, network blip, session timeout. System should handle gracefully.
🔴 Negative Path Invalid intent or unauthorized actor: wrong permission, bypass attempt, malformed input. System should block/protect.

🟢 All Happy Paths

6 happy path scenarios across all 5 processes.

🟢 Happy Path Happy Path 1: Full lifecycle — Draft to PublishedProcess 01
Persona: All approval-chain personas (NOFO Dev → Program Office → OA concurrers → OA Leadership → OST Admin → DAS → S1)  ·  Runs in: All 6 environments (run separately per environment per protocol)
Preconditions
  1. Test users for each persona exist with correct PSGs assigned
  2. Pre-defined GFA data templates are loaded (Construction and Non-Construction criteria)
  3. Grants.gov test environment is reachable and credentials valid
  4. No prior in-flight test FOs from a previous run (clean slate or staging cleared)
Steps
  1. NOFO Dev Team logs in and creates a new Funding Opportunity record from the Funding Opportunity tab
  2. Complete the FO Creation screen with all required fields (Title, Type Code, Program Type, Funding Amount, etc.) and Save
  3. Program Office opens the recently created NOFO record and completes NOFO Setup tab with valid staging data across all 11 sections:
    1. NOFO Identity & Announcement
    2. Funding Details & Description
    3. Application Timeline
    4. Eligibility & Program Description
    5. Application Contents & Format
    6. Submission & Intergovernmental Review
    7. Award Notices & Other Information
    8. Evaluation & Review Setup
    9. Review Factors
    10. Risk Criteria
    11. Intake and Eligibility Requirements
  4. Program Office clicks "Submit for Approval" — confirms FO advances to Under OA Review status
  5. Grant Office concurrer (notified via email) opens the FO, navigates to the Concurrence/Approval tab, and marks Grant Office concurrence
  6. OA Legal concurrer marks legal concurrence on the same tab
  7. OA CFO concurrer marks CFO concurrence
  8. OA Leadership opens the FO once all upstream concurrences are complete and clicks "OA Review Complete" — FO advances to Under OST-P Review
  9. OST Admin reviews the FO and marks OST-P concurrence (or routes to DAS for concurrence)
  10. DAS opens the FO and marks DAS concurrence on the Concurrence/Approval tab
  11. S1 reviews the FO and clicks "S1 Approval Complete" — FO advances to S1 Approval Complete status
  12. Program Office clicks "Publish Funding Opportunity" — status moves to Pending Publish
  13. After refresh, status changes to Published
  14. Verify the FO, related documents, and packages populate in Grants.gov test environment
Expected
  • Each status transition is recorded with timestamp and acting user on the Approval History
  • Each concurrence checkbox shows the concurrer's name and date
  • Email notifications fire to the next persona at each gate
  • Once published, the Synopsis Tab shows Version Number = 1
  • The FO is searchable in Grants.gov test environment with correct metadata
🟢 Happy Path Happy Path 2: Successful Grants.gov submission → Salesforce Application Form recordProcess 02
Persona: Applicant / External (origin) and System / API (intake)  ·  Runs in: All 6 environments
Preconditions
  1. Published FO exists and is open for applications (within Application Open/Close window)
  2. Applicant has a valid SAM-registered UEI
  3. Grants.gov → Salesforce intake integration is configured (TC 225, 214)
  4. Apex Pub/Sub API is enabled (GRAN-244)
Steps
  1. Applicant logs into Grants.gov and locates the published FO using the FO Number
  2. Applicant downloads the Application Package and completes all required forms (SF-424, project narrative, budget)
  3. Applicant submits the completed package via Grants.gov Workspace
  4. Salesforce intake job (or "Retrieve Applications" button) pulls submission metadata into Salesforce
  5. System creates an Application Form record in Salesforce with all SF-424 fields populated (TC 111)
  6. System runs SAM/UEI deduplication check against existing applicants
  7. Application appears in the FO's Related → Application Forms list
  8. Program Office can navigate to the Application Form record and verify all data
Expected
  • Application Form record exists in Salesforce with status Submitted
  • All key SF-424 fields are populated (UEI, applicant name, requested amount, project title)
  • Submission timestamp matches Grants.gov submission time
  • No duplicate Application Form records created for the same applicant on the same FO
  • Account record exists or is linked correctly via UEI
🟢 Happy Path Happy Path 3: Compliance + Merit + Combined Rating end-to-endProcess 03
Persona: Application Reviewer (assigned to the FO) and Program Office  ·  Runs in: All 6 environments
Preconditions
  1. FO is published and applications have been received and intake-processed
  2. Reviewer is assigned to the FO with the GL-Application Reviewer PSG (GRAN-263)
  3. Review Factors and Risk Criteria are configured on the FO (TC 181-185)
  4. Reviewer is logged in and notification email received
Steps
  1. Reviewer navigates to the assigned Application Form record from email link
  2. Opens the Compliance Review record under the Related tab (TC 250)
  3. Under the Review tab, completes Evaluation (Merit) Criteria for each criterion:
    1. Technical Merit — assess project quality and feasibility
    2. Broader Impacts and Benefits — assess wider effects
    3. Project Readiness (for construction projects)
    4. Project Benefits (for construction projects)
  4. For each criterion, selects rating (Highly Recommend / Recommend / Acceptable / Not Recommended) and adds Reviewer Comment
  5. Completes Selection Considerations section, including COI/NDA acknowledgment (TC 265)
  6. Clicks Save Progress to persist responses
  7. Clicks Submit when review is complete
  8. Status moves to one of the 17 valid review statuses (TC 223)
  9. Program Office views Combined Rating once all assigned reviewers complete
Expected
  • All review responses persist correctly across save/submit cycles
  • Cannot submit until all required criteria are rated
  • Combined Rating uses correct weight per GRAN-260 (point weight = 100 for primary criteria)
  • Submitted review record is read-only to the reviewer after submission
  • Audit log captures reviewer name, timestamp, and final rating
🟢 Happy Path Happy Path 4: SRT → Concurrences → S1 Approval → Notify ApplicantsProcess 04
Persona: Program Office through S1, then external applicants  ·  Runs in: All 6 environments
Preconditions
  1. All applications for the FO have completed review (Combined Ratings finalized)
  2. Selection Recommendation Table (SRT) data is populated
  3. Award memo template is configured
  4. All approval personas have correct PSGs
Steps
  1. Program Office opens the SRT (Data for RAAW Report — TC 239)
  2. Reviews proposed selections and award amounts
  3. Generates the Award Memo / RAAW document
  4. Submits the memo for OA concurrence
  5. Grant Office concurrer reviews and signs off
  6. OA Legal concurrer reviews and signs off
  7. OA CFO concurrer reviews and signs off
  8. OA Leadership reviews and approves
  9. S1 reviews the consolidated award memo and approves (status moves to Award Selections Approved by S1)
  10. Program Office returns to the FO and clicks "Notify Applicants" (TC 262)
  11. System presents a table of all applicants with selection status
  12. Program Office reviews the table and confirms notifications
  13. Notifications fire to all applicants (selected and non-selected)
Expected
  • Selected applicants receive award notification emails with next-steps and award amount
  • Non-selected applicants receive non-selection emails with feedback option
  • All notifications are logged in Salesforce Activity history
  • Audit log captures S1 approval timestamp and acting user
  • No duplicate notifications sent
🟢 Happy Path Happy Path 5: Material Modification → Republish (TC 203)Process 05
Persona: Program Office + OA approval chain  ·  Runs in: All 6 environments
Preconditions
  1. FO is in Published status with applications already received
  2. Material change is needed (e.g., extended deadline, amended eligibility)
Steps
  1. Program Office opens the published FO
  2. Initiates Material Modifications workflow (Step 3 — TC 203)
  3. Edits the affected fields (Application Close Date, Eligibility Description, etc.)
  4. Routes the modified FO through OA concurrences again
  5. OA concurrers and OA Leadership re-approve the modifications
  6. OST Admin re-runs OST-P review on the modified FO
  7. S1 re-approves
  8. Program Office clicks "Re-Initiate Approval" on the Republished NOFO (TC 203 second variant)
  9. Status advances; system republishes to Grants.gov
  10. Synopsis Tab Version Number increments to 2
Expected
  • Modification is captured with full audit trail
  • Original FO history is preserved (not overwritten)
  • Existing applications remain valid; applicants notified of changes
  • Grants.gov reflects the updated FO with new version number
🟢 Happy Path Happy Path 6: COI/NDA acknowledgment on Application Account (TC 265)Process 05
Persona: App Reviewer  ·  Runs in: All 6 environments
Preconditions
  1. Reviewer is assigned to an Application Form
  2. Account record (the applicant) has COI/NDA tracking fields
  3. TC 265 functionality deployed (Account COI/NDA Checkbox and Upload)
Steps
  1. Reviewer opens the Account associated with the Application
  2. Reviews the COI/NDA section
  3. If a conflict exists: checks the COI checkbox and adds explanatory comment
  4. If NDA is required: uploads signed NDA document
  5. Saves the Account record
Expected
  • Checkbox state and uploaded NDA persist on the Account record
  • COI flag is visible on subsequent reviews of any application from the same applicant
  • Audit log captures the acknowledgment with reviewer and timestamp
  • Reviewer can proceed with Compliance Review only after COI/NDA acknowledgment

🟡 All Sad Paths

12 sad path scenarios across all 5 processes.

🟡 Sad Path Sad Path 1: Required field missing on NOFO SetupProcess 01
Persona: Program Office  ·  Runs in: All 6 environments
Preconditions
  1. Program Office user has access to a Draft FO
  2. NOFO Setup tab is open
Steps
  1. Program Office attempts to leave a required field blank (e.g., Total Amount of Funding)
  2. Click Save
Expected
  • Save is blocked with a validation error identifying the missing required field
  • FO does not advance status
  • No data is persisted for the partial entry
  • User can correct the field and save successfully on retry
Defects to log if seen:
  • Save succeeded with empty required field
  • Validation error message is unclear or doesn't identify which field
  • FO advanced status despite incomplete data
🟡 Sad Path Sad Path 2: Concurrer rejects and sends backProcess 01
Persona: OA Legal (or any OA concurrer)  ·  Runs in: All 6 environments
Preconditions
  1. FO is in Under OA Review status with at least Grant Office concurrence already granted
  2. OA Legal user has access to the FO
Steps
  1. OA Legal opens the FO from the email notification
  2. Reviews content and identifies an issue (e.g., missing eligibility statement)
  3. Adds a Chatter comment describing the issue
  4. Does NOT mark OA Legal concurrence; instead, sends an email/Chatter notification back to Program Office
  5. Program Office receives notification, opens the FO, makes corrections
  6. Program Office re-saves and re-routes for OA review
Expected
  • FO remains in Under OA Review until concurrer signs off
  • OA Leadership cannot approve while OA Legal concurrence is missing
  • Chatter comment is visible on the FO record with timestamp
  • After Program Office corrections and re-routing, OA Legal can re-review and concur
Defects to log if seen:
  • OA Leadership Approval button became enabled before all concurrers signed off
  • Chatter comment did not save or wasn't visible to other personas
🟡 Sad Path Sad Path 3: Publish to Grants.gov fails — connectivity issueProcess 01
Persona: Program Office  ·  Runs in: All 6 environments (most likely surfaces in DTC UAT or Production due to live integration)
Preconditions
  1. FO is in S1 Approval Complete status
  2. Grants.gov test endpoint may be unreachable or credentials expired
Steps
  1. Program Office clicks "Publish Funding Opportunity"
  2. Status moves to Pending Publish
  3. Refresh after 30 seconds
Expected
  • If integration fails, status remains Pending Publish and an error notification appears

  • FO does not advance to Published until integration succeeds
  • Sys Admin can re-trigger publish via Apex job once Grants.gov is reachable
Defects to log if seen:
  • Status incorrectly advanced to Published despite Grants.gov integration failing
  • No error notification surfaced to Program Office
  • Retry mechanism is not available or is hidden

🟡 Sad Path Sad Path 5: Malformed XML payload from Grants.govProcess 02
Persona: System / API + Sys Admin  ·  Runs in: All 6 environments (most likely surfaces in DOT QA / DTC QA during integration testing)
Preconditions
  1. Grants.gov submission contains malformed or incomplete XML (e.g., missing required namespace, truncated element)
  2. Apex intake handler is configured with error capture
Steps

  1. FO is Published

Steps
  1. Non-PO user attempts to initiate Material Modification (clicks Edit on a locked field, or invokes the Material Modification flow)
Expected
  • Material Modification workflow is restricted to Program Office personas
  • Edit attempt by other personas is blocked at the field-security or button-visibility layer

Defects to log if seen:
  • Non-PO user successfully initiated a Material Modification
  • Field permissions inconsistent with workflow permissions
